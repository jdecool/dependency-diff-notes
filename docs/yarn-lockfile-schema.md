# yarn.lock wire format reference

Research notes for the Yarn lockfile parser (`internal/yarnlock`, see ADR 0004).
Primary source: Yarn's own parser grammar, [`yarnpkg/berry: packages/yarnpkg-parsers/sources/grammars/syml.pegjs`](https://github.com/yarnpkg/berry/blob/master/packages/yarnpkg-parsers/sources/grammars/syml.pegjs) - the literal parser code (`@yarnpkg/parsers`), which handles both formats below and is more authoritative than Yarn's docs site (`yarnpkg.com/advanced/lockfile-format` was unreachable at research time).
Cross-checked against real `yarn.lock` files pulled from GitHub (facebook/react, webpack/webpack, and others).

Neither format records a production-vs-development distinction per package - that information lives only in `package.json`, which this package does not read (see CONTEXT.md: Production dependencies / Development dependencies).
A Lock returned by `Parse` is therefore always `Combined`.

## Classic (v1)

Header, verified byte-for-byte identical across every real Classic lockfile sampled:

```
# THIS IS AN AUTOGENERATED FILE. DO NOT EDIT THIS FILE DIRECTLY.
# yarn lockfile v1
```

**Spec keys**: comma-space (`, `) separated when one resolution satisfies multiple requested ranges.
Quoting follows the grammar's `pseudostringLegacy` production (an unquoted token may start with a letter, digit, or `/`, and may not contain a space, `:`, or `,`):

- Unscoped, no special characters -> unquoted: `accepts@~1.3.4, accepts@~1.3.5:`
- Scoped (`@`-prefixed) -> always quoted, since `@` cannot start an unquoted token: `"@babel/code-frame@^7.0.0", "@babel/code-frame@^7.12.13":`
- Anything containing `:` (git URLs) -> always quoted: `"libsignal@git+https://github.com/adiwajshing/libsignal-node.git":`

A parser must accept both quoted and unquoted forms regardless - it's a compatibility rule baked into the same grammar Berry uses.

**Entry fields**, 2-space indent per level (`INDENT_STEP = 2` in the grammar):

```
<key(s)>:
  version "<semver>"
  resolved "<url>[#<sha>]"
  integrity <sha512-...>        # absent for git dependencies
  dependencies:
    <name> "<range>"
```

**Git dependency**, two real forms observed:

- GitHub shorthand `user/repo#tag` -> `resolved` becomes a codeload tarball URL ending in the full 40-character resolved commit SHA, and there's no `integrity` field:
  ```
  tooling@webpack/tooling#v1.26.4:
    version "1.26.4"
    resolved "https://codeload.github.com/webpack/tooling/tar.gz/978dc1c9680ef7d79f5f5c02c3439385d7937c39"
  ```
- Explicit `git+https://` protocol -> `resolved` keeps the protocol prefix, followed by `#<full-commit-sha>`, again no `integrity`:
  ```
  "libsignal@git+https://github.com/adiwajshing/libsignal-node.git":
    version "2.0.1"
    resolved "git+https://github.com/adiwajshing/libsignal-node.git#11dbd962ea108187c79a7c46fe4d6f790e23da97"
  ```
  `version` here is whatever `package.json` at that commit declares - not derived from the requested ref.

A registry package's `resolved` tarball URL *also* carries a trailing `#<hash>` fragment (a checksum, historically redundant with `integrity`), which must not be mistaken for a git reference: it never changes independently of the published version.
Since registry tarballs are always `.tgz` (not `.tar.gz`), `internal/yarnlock` distinguishes the two by checking for a `git+`/`git://` prefix or a `/tar.gz/` codeload path segment, not by the presence of a `#` fragment alone.

## Berry (v2+, real YAML via the same "syml" grammar)

Header:

```
# This file is generated by running "yarn install" inside your project.
# Manual changes might be lost - proceed with caution!
```

**Top-level**: a flat map at the document root, spec keys comma-joined exactly as in Classic (the grammar's `PropertyStatement` production splits keys on `,`).
`__metadata` is a sibling top-level key, not a package - it must be skipped when iterating:

```yaml
__metadata:
  version: 10
  cacheKey: 10
```

(the lockfile-format version observed varies - 4, 8, 10 - across sampled repos; it is not the Yarn release version.)

**Package entry fields** - `version`, `resolution`, `dependencies` (optional), `checksum` (absent for `workspace:` entries), `languageName` (`node` or `unknown`), `linkType` (`hard` or `soft`):

```yaml
"@algolia/autocomplete-core@npm:1.17.9":
  version: 1.17.9
  resolution: "@algolia/autocomplete-core@npm:1.17.9"
  dependencies:
    "@algolia/autocomplete-shared": "npm:1.17.9"
  checksum: 10/cf4f0f1d9e0ca4e7...   # opaque; treat as such, format varies by metadata version
  languageName: node
  linkType: hard
```

`resolution` is the canonical pinned identity `<name>@<protocol>:<resolved-locator>` - it is *not* a copy of the header key (the header key can carry ranges; `resolution` always carries the resolved locator).
`version`, notably, is its own top-level field on every entry (registry or git) - it does not need to be parsed out of `resolution`.

`linkType: soft` marks a workspace-local package (no physical install, `languageName: unknown`) - skipped the same way as npm's `"link": true`, confirmed via a real example: `"@yarnpkg/cli@workspace:packages/yarnpkg-cli"` with `linkType: soft`.

**Git dependency**, real example ([fossas/fossa-cli test/Yarn/V2/testdata/yarn.lock-example](https://github.com/fossas/fossa-cli/blob/master/test/Yarn/V2/testdata/yarn.lock-example) - a test fixture, but shaped consistently with the grammar and real `yarn install` output; no genuine non-fixture Berry lockfile with a git dependency was found among the popular repos sampled):

```yaml
"underscore@jashkenas/underscore#tag=1.13.1":
  version: 1.13.1
  resolution: "underscore@https://github.com/jashkenas/underscore.git#commit=cbb48b79fc1205aa04feb03dbc055cdd28a12652"
  checksum: 560609fdb4ba2c30e79db95ea37269982d1a2788d49b78f0de4f391da711bc2495d5fbddd6d24e7716fccf69959e445916af83eb5de1ad137b215777e2d32e4d
  languageName: node
  linkType: hard
```

Unlike Classic, Berry git dependencies *do* carry a `checksum`.
The commit lives in `resolution`, after `#commit=` (full 40-character SHA) - `internal/yarnlock` extracts it from there, never from `version`.

## Open ambiguities

- `checksum`'s format is inconsistent across lockfile-format versions (plain hex vs. `<cacheKey>/<hex>`) - `internal/yarnlock` never reads it, treating it as wholly opaque.
- No genuine (non-test-fixture) Berry lockfile with a git dependency was found among the repos sampled; the shape above is inferred to be representative but wasn't cross-checked against a second independent source the way the rest of this document was.
